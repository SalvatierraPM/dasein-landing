# Design Principles

This repository is not trying to look like a generic SaaS starter or a speculative art project. The site needs to signal consulting seriousness while still feeling original.

## Principles

### 1. Lead with the offer, not the effect

Motion and visual texture support the narrative, but the page still has to answer the basic public questions quickly:

- What is Dasein?
- Who is it for?
- What does it sell?
- Why should anyone pay attention?

### 2. Avoid fake maturity

The site should feel credible without pretending there is already a deep public case-study archive or a fully proven platform behind it.

### 3. Use atmosphere as framing, not decoration

Particles, depth, and scroll choreography are there to create a systems-oriented mood. They should reinforce the idea of signals, actors, and complexity, not overpower the copy.

### 4. Stay Spanish-first in product voice

The public site copy is written for the current audience and market context. The repository documentation can stay more global and technical, but the site itself does not need to flatten its voice for generic internet neutrality.

### 5. Keep the code legible

This repo works better as a public artifact when the implementation reads like deliberate craft rather than a component dump. That means:

- descriptive sections
- explicit layout structure
- limited conceptual sprawl
- avoiding one-off experimental files in the root

### 6. Separate presentation from proof

The landing page can present the method and venture direction, but proof should arrive through future case studies, public methodology notes, and documented results. The repo should make that boundary obvious.
